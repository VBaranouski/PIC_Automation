2. Objectives

The main objectives of this release candidate are to introduce Delegated Requirements Traceability and improve Data Protection & Privacy workflows in PICASso, along with a set of defect fixes across BackOffice, requirements management, release history, and the Cybersecurity Residual Risks tab.

3. New Features and Enhancements

3.1. Delegated Requirements Traceability

Users can now fully trace requirements that have been delegated to other products. Delegation details — target product, release, requirement, due date, and delegated part — can be entered manually, in bulk, or via XLSX file upload. Details are visible across the UI, synchronized with Jira/Jama, propagated to cloned releases, and logged in release history.

Manual and Bulk Delegation

  1. Delegation fields in Edit Requirement pop-up — On the Process/Product Requirements tab, clicking the three-dot menu next to a requirement and selecting “Delegated”, or using “View/Edit > Edit”, now shows additional delegation fields in the pop-up:
    • Product Name (Dropdown, mandatory) — search by Product ID or Product Name; includes “Other Product” option pinned to top for products not found in PICASso.
    • Other Product (Free text, mandatory if “Other Product” selected).
    • Contact Person (Lookup, mandatory if the product does not exist in PICASso) — the person responsible for requirement implementation in the delegated-to product.
    • Product Release (Dropdown / Free text, optional) — lists releases of the selected product with an “Other Release” option; free text if “Other Product” was selected.
    • Requirement (Dropdown, optional) — filtered by requirement type (process/product); includes search by Code or Name and a “Requirement Not Found” option.
    • Status (Read-only, dynamic) — current status of the delegated-to requirement; updates in real time.
    • Evaluation Status (Read-only, dynamic) — evaluation status of the delegated-to requirement; updates in real time.
    • Due Date (Date picker, optional) — target date for completing the delegated requirement.
    • Delegated Requirement Part (Free text, optional) — specifies which part of the current requirement is covered by the delegated-to requirement.
  2. Multiple products and requirements — A “+Add Product” button adds additional delegation forms. A “+Add Requirement” button adds additional requirement fields within a product. Both have “Remove” buttons (bin icon) with confirmation pop-ups; at least one empty form/field always remains.
  3. Due Date confirmation — If Due Date is empty on save, a confirmation pop-up is shown: “You haven’t specified a due date for requirement(s) delegated to following products: <product name>. Are you sure you want to save requirement without setting it?”
  4. Bulk delegation — Selecting multiple requirements and using Bulk Actions “Edit” displays the same delegation form when “Delegated” status is chosen. All selected requirements are updated simultaneously.
  5. Overwrite confirmation — If any selected requirements already have delegation details, a confirmation is shown before overwriting: “Requirements <code 1>, <code 2>, ... already have delegation details. They will be overwritten with new details you specified. Do you wish to proceed?”

Delegation Details Display

  1. Requirement Info pop-up (view mode) — Displays the following delegation fields in read-only view for the delegating requirement: Delegated to Product, Delegated to Release, Delegated to Requirement, Status (dynamic), Evaluation Status (dynamic), Delegated Requirement Part, Due Date. All values are clickable, leading to the linked entity in PICASso in read-only mode.
  2. Edit Requirement pop-up (edit mode) — All delegation fields are editable, allowing users to update or change delegation targets at any time.
  3. Information icon on the delegated-to requirement — The status of a requirement that has been delegated to it is highlighted with an information icon and tooltip: “Requirement(s) was delegated to this requirement.”
  4. Collapsible “Delegated Requirement” section — The Requirement Info pop-up of the delegated-to requirement includes a collapsible section showing: Requirement Name, Status, Delegated from Product, Delegated from Release, Comment, Due Date. If the delegating requirement’s status changes away from “Delegated”, the entry is removed from this section.
  5. Delegation details on Cybersecurity Residual Risks (CSRR) tab — Delegation information is shown in the View Requirement and Edit Evaluation Status pop-ups within SDL Processes Summary and Product Requirements Summary sections:
    • Delegating requirement: Delegated to Product, Release, Requirement, Status, Evaluation Status, Delegated Requirement Part, Due Date.
    • Delegated-to requirement: collapsible section with Requirement Name, Status, Delegated from Product, Delegated from Release, Comment, Due Date.
  6. Delegation propagation on release clone — “Delegated to” details and “Delegated From” information are both copied to the cloned release.

File Upload via XLSX (Export – Import)

  1. New “Delegation Details” tab in exported XLSX — The exported requirements file now includes an additional tab with: Requirement Code, Requirement Name, Delegated to Product ID (format: PIC-xxx), Delegated to Product Name, Delegated to Release ID, Delegated to Product Release, Delegated to Requirement Code, Delegated to Requirement Name, Delegation Due Date (YYYY-MM-DD, optional), Delegated Requirement Part (optional). If delegation details were previously entered, the tab is pre-populated on export.
  2. Import validation — Product ID format, Product Name (mandatory if ID unknown), Requirement Code (must exist in library and be scoped for the release), Due Date format, missing delegation details for “Delegated” status rows, duplicate Product+Release+Requirement combinations, and deactivated Product or Release references are all validated with inline error messages showing affected line numbers.
  3. Guidance on Instructions tab — The Instructions tab in the exported file provides step-by-step guidance for all fields, format requirements, and examples for multi-product and multi-requirement delegation entries.

Jira / Jama Synchronization and History

  1. Warning for delegated requirements synced from Jira/Jama — If a requirement’s status is changed to “Delegated” via Jira/Jama and synchronized with PICASso, a warning icon with the tooltip “Please specify delegation details” appears next to the requirement. If at least one such warning exists, a notification is shown when submitting the release to the next stage (submission is not blocked).
  2. Delegation history logging — A new “Delegation Details Update” activity is recorded in the release history when delegation details are added, changed, or updated on a delegated-to requirement (“Delegated from” fields). The activity is available as a filter option in the “Filter by Activity” dropdown.

3.2. Data Protection & Privacy — Expected Maturity Level Display

The Expected Maturity level is no longer shown as a default value in the Privacy Section sidebar, allowing users to clearly distinguish sections that have already been reviewed from those that have not yet been evaluated.

Key Features

  1. “Not Yet Evaluated” default stamp — When only the Expected Maturity level is defined for a Privacy Section (no Maturity level has been selected manually yet) and the section is applicable for the release (not marked as “Not Applicable”), the section now displays the stamp “Not Yet Evaluated” by default, instead of showing the Expected Maturity value.
  2. Dynamic maturity stamp after manual review — As soon as a reviewer manually selects and saves a Maturity level for a section, the stamp updates to display the selected maturity level name. The stamp also reflects the relationship between the manually selected value and the Expected Maturity:
    • Manually selected > Expected Maturity — stamp shows an upward arrow.
    • Manually selected < Expected Maturity — stamp shows a downward arrow.
    • Manually selected = Expected Maturity — stamp shows no arrow.

3.3. Defect Fixes

The following defects were fixed with this release:

  • Requirements duplicated in release and submitted to Jama as duplicates
    Fixed an issue where requirements appeared duplicated in a release and were consequently submitted to Jama as duplicate entries.
  • Users with limited scope could see Organizations beyond their permitted access in filters
    Fixed an issue where users with a limited organizational scope were able to see Organizations in filters beyond their assigned access level.
  • Role of approving user missing in release history for multi-approver stages
    Fixed an issue where the approving user’s role was not recorded in the release history for stages requiring approval from multiple stage responsibles (e.g. the Dedicated Privacy Advisor approving the Review & Confirm stage).
  • Data Protection & Privacy Risk not cloned to the new release
    Fixed an issue where the Data Protection & Privacy Risk value was not copied when cloning a release. The DPP Risk from the original release is now correctly carried over to the cloned release. If the DPP Risk was updated on the Review & Confirm stage, the updated value is cloned.
  • Error when editing actions if a closed action has empty mandatory fields
    Fixed an issue where the system returned a “Review mandatory fields” error and highlighted mandatory fields in a closed action when the user attempted to edit or save another action in the same section. This occurred for actions created before the Actions Management update, when the evidence link was not yet mandatory for the closed status. Closed actions are no longer subject to mandatory field validation.
  • Must/Should column missing in exported Product Requirements XLSX
    Fixed an issue where the Must/Should column was absent from the Data tab of the exported Product Requirements XLSX file. The column is now included in the export.
  • Summary text in CS Residual Risks section displayed in a single truncated line
    Fixed an issue where the text in the Summary field of the SDL Processes Summary section on the Cybersecurity Residual Risks tab was displayed in a single truncated line instead of the full content.
  • [BackOffice] Source List — Incorrect error message for invalid Excel template
    Fixed an issue where uploading an Excel file with an incorrect columns structure to BackOffice Source List displayed a misleading error referencing a column that did not exist in the file. The error message is now: “Columns structure is incorrect, please use the columns from the template file.”
  • [BackOffice] API Extraction/Ingestion Configuration — Inactive records cannot be activated
    Fixed an issue where attempting to re-activate an inactive Consumer record in BackOffice API Extraction/Ingestion Configuration returned an “Error executing query” error. Inactive records can now be activated successfully.
  • [BackOffice] SCA Tool List — Severity cannot be added
    Fixed an issue where clicking “Add Severity Input” in BackOffice SCA Tool List triggered the error “Cannot read properties of null (reading ‘getData’)”. Severity entries can now be added without errors.
