AI-driven test automation at this scale isn't without friction. Here's an honest account of the real challenges — how they surface, why they exist, and how we're working around them.

Known Constraints

Three Structural Challenges

These aren't gaps in the tool — they're constraints imposed by the AI infrastructure, the application environment, and the network architecture. All are acknowledged and actively mitigated.

memory
construction Actively Managed

AI Token Consumption

PICASso's complexity means working with large data volumes — 4,220+ test cases, multi-layer page objects, OutSystems DOM snapshots, Playwright traces, and multi-file spec contexts. Complex prompts that load all of this simultaneously exhaust AI quotas rapidly.

Where it bites: batch scenario analysis, large spec-file generation, multi-scenario exploratory runs with Playwright MCP, and sessions involving both test writing and locator validation in a single context window.

Current mitigation: scope work per workflow or feature area rather than full-suite sessions; use targeted queries (--workflow, --area) to keep context minimal; split preparation and scripting into separate AI sessions.

construction Partially Mitigated

Environment Performance & Stability

The QA environment runs on OutSystems, which can be slow under load. Response times are unpredictable — page transitions, partial refreshes, and data-heavy screens can stall for several seconds, causing otherwise correct tests to fail on timeouts rather than logic.

Root cause: OutSystems server-side rendering and partial-refresh patterns don't expose reliable network-idle events, making standard "wait for load" strategies insufficient. Heavy GRC data (documents, roles, release histories) amplifies the problem.

Current mitigation: custom waitForOSScreenLoad() and waitForTableData() helpers replace generic timeouts; flaky tests are annotated with test.slow() and investigated for environment-side root causes before weakening assertions.

block Structural Constraint

SE Network Access — No External Reachability

The PICASso application is only accessible from within the SE internal network. There is no pathway for external services — cloud CI runners, hosted web apps, or external AI agents — to reach the application directly.

Impact: the tracker tool cannot be deployed as a hosted web application accessible to external team members, because any test execution triggered from outside the SE network would have no access to the platform under test.

Current approach: the tracker runs as a local development server within the SE network; CI execution is scoped to SE-hosted runners or engineer workstations with VPN access. Web deployment is planned for when network access is resolved.
Transparency Over Perfection

These constraints are real and have shaped how the automation pipeline is designed. The token budget drives the decision to work per-workflow rather than full-suite. Environment instability drives the investment in OutSystems-specific wait helpers. Network access drives the local-first deployment model. Each pitfall has an engineering response — and each response is encoded directly into the picasso-test-automation skill so AI agents handle them automatically.
