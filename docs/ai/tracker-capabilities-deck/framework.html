The Test Automation
Framework

Engineering choices, folder structure, naming conventions, and best practices behind the PICASso automation suite. Built on Playwright + TypeScript, with a four-layer architecture and Allure reporting — designed for scale, maintainability, and AI-driven extension.

The Stack

Playwright + TypeScript + Allure

Three deliberate choices — each one solves a specific problem that alternatives don't.

stadia_controller

Playwright — The Runtime

A single library that drives Chromium, Firefox, and WebKit through the browser's own CDP / DevTools protocol. No WebDriver proxy, no flaky JSON-over-HTTP bridge.

What it does: auto-waiting web-first assertions, getByRole / ARIA-aware locators, network interception, headed/headless execution, trace viewer with DOM snapshots, video recording on failure, parallel worker execution, and a first-class TypeScript API.

code_blocks

TypeScript — The Language

Strict-mode TypeScript with path aliases (@pages/, @locators/, @fixtures/, @helpers/), union types for test data, and ReturnType<typeof fn> for locator factories.

Why it matters: compile-time catches for locator typos, refactor-safe page objects, IDE autocomplete for every POM method, and one language for tests, fixtures, tracker UI, and backend scripts.

analytics

Allure — The Reporting

Rich HTML reports with suites, severity, tags, step-by-step breakdowns, screenshots, traces, and trend history across runs.

Why Allure: the built-in allure-playwright reporter plus every test requiring allure.suite(), allure.severity(), allure.tag(), and test.step() blocks → reports read like a narrative, and the tracker parses the summary JSON to sync execution status automatically.

Four-Layer Rule

Separation of Concerns

The strict layering prevents the two classic Playwright antipatterns: locators leaking into tests, and page objects containing raw page.locator() strings.

Locator Factorysrc/locators/*.locators.ts
Pure selector functions. Each file exports a factory that takes a Page and returns a typed bag of locators. Zero logic — no waits, no clicks, no assertions. Type derived via ReturnType<typeof fn>.
Page Objectsrc/pages/*.page.ts
Extends BasePage. Holds the locator bag as private readonly l. Exposes semantic actions (login(), selectProduct()) and assertion helpers (expectLoggedIn()). Never contains a raw page.locator() call.
Fixturesrc/fixtures/*.fixture.ts
Composable test fixtures that instantiate page objects and inject shared state (auth storage, user credentials, DOC-state). Tests import test from @fixtures/index, not from @playwright/test.
Spec Filetests/<area>/*.spec.ts
describe blocks, test() cases, test.step() narrative, and Allure metadata. No inline locators, no raw expect() on page elements — all verification flows through POM assertion methods.
check_circle Why this matters at scale

With 4,220+ scenarios and an evolving OutSystems UI, a single locator change (e.g. a role rename on a button) must propagate through the codebase without touching tests. The locator-factory layer means one edit in one file cascades to every spec that uses it — a property no flat structure delivers.

This is also what makes the AI reliable: when the create-auto-tests skill generates tests, it's producing code into four well-defined slots, not freeform scripts.

Example

One Scenario, Four Files

A minimal login test — showing exactly where each piece of code lives. 

// 1. src/locators/auth.locators.ts — selectors only
import { Page } from '@playwright/test';

export const authLocators = (page: Page) => ({
  emailInput:    page.getByLabel('Email'),
  passwordInput: page.getByLabel('Password'),
  submitButton:  page.getByRole('button', { name: 'Sign In' }),
  errorAlert:    page.getByRole('alert'),
});
export type AuthLocators = ReturnType<typeof authLocators>;
// 2. src/pages/login.page.ts — actions & assertions
import { Page, expect } from '@playwright/test';
import { BasePage } from '@pages/base.page';
import { authLocators } from '@locators/auth.locators';

export class LoginPage extends BasePage {
  readonly url = '/login';
  private readonly l;

  constructor(page: Page) { super(page); this.l = authLocators(page); }

  async login(email: string, password: string): Promise<void> {
    await this.l.emailInput.fill(email);
    await this.l.passwordInput.fill(password);
    await this.l.submitButton.click();
  }

  async expectErrorMessage(message: string): Promise<void> {
    await expect(this.l.errorAlert).toContainText(message);
  }
}
// 3. src/fixtures/index.ts — inject POM into tests
import { test as base } from '@playwright/test';
import { LoginPage } from '@pages/login.page';

export const test = base.extend<{ loginPage: LoginPage }>({
  loginPage: async ({ page }, use) => { await use(new LoginPage(page)); },
});
export { expect } from '@playwright/test';
// 4. tests/auth/login.spec.ts — narrative only
import { test, expect } from '@fixtures/index';
import * as allure from 'allure-js-commons';

test.describe('Authentication @smoke', () => {
  test('AUTH-LOGIN-001 — invalid credentials show error', async ({ loginPage }) => {
    await allure.suite('Authentication');
    await allure.severity('critical');
    await allure.tag('smoke');

    await test.step('Attempt login with wrong password', async () => {
      await loginPage.goto();
      await loginPage.login('user@sequoia.com', 'wrong-password');
    });

    await test.step('Verify error is shown', async () => {
      await loginPage.expectErrorMessage('Invalid credentials');
    });
  });
});

Locator Priority

Always use the first option that works. Semantic accessibility locators come first — they mirror how users actually find elements on the page.

Timing

Wait Strategy — OutSystems-Aware

OutSystems keeps WebSockets alive, so networkidle never fires. Arbitrary sleeps are forbidden. Every wait must key off an observable DOM readiness signal.

SignalWhen to Use
Heading visibleAfter navigation — the heading is the page-ready contract
OSUI overlay goneAfter screen navigation or data fetch — waitForOSScreenLoad()
Table headers visiblePages that render data asynchronously — waitForTableData()
Control becomes enabledForms that disable inputs until dependency loads
Save button appearsAfter submit — wait for "Edit Product" to appear, not URL change
Banned waitForTimeout()Use explicit element assertions
Banned networkidleOutSystems WebSocket keeps it alive forever

Engineering Choice

Why TypeScript, Not Python

Both Playwright Python and Playwright TypeScript exist. For a project the scale of PICASso — 4,220+ scenarios, an AI-generated codebase, and a custom tracker UI — TypeScript wins on every axis that matters.

TypeScriptPython
Type safety Native, strict mode, compile-time errors before run Runtime type hints only — catches errors at test-time
Playwright API parity First-class — every feature ships in TS first Port — features can lag Node API by weeks
Async model Native async/await — matches browser event loop Requires asyncio ceremony; sync API exists but is slower
Language of web Same language as the app under test, fixtures, tools, tracker UI Context switch between Python tests and TypeScript frontend
IDE refactoring Rename-across-codebase, find-all-references, safe inline Dynamic typing limits IDE certainty on refactors
Locator typing ReturnType<typeof fn> derives a typed locator bag No equivalent pattern — dicts or dataclasses
AI code generation Strong type signatures guide the model — fewer hallucinations Duck typing lets the model invent method names silently
Allure support allure-playwright reporter + allure-js-commons allure-pytest works but with a different API surface
Tracker / tooling Single language — tracker, scripts, specs, fixtures all TS Would require a second language for the tracker UI
lightbulb The deciding factor

This is an AI-generated codebase. Strong static types make the AI skill dramatically more reliable — the compiler rejects invented methods, mistyped parameters, and incorrect locator signatures before a test ever runs. Python's runtime typing would push the same errors to test-time, where they manifest as flakes.

Additionally, the tracker UI (Express + TypeScript), the scripts (tsx), and the tests share one toolchain. One install, one type system, one mental model.

Non-Negotiables

Framework Best Practices

Every test generated or reviewed must pass this checklist. These rules are encoded in the create-auto-tests AI skill — and verified manually in code review.

rule

Strict Four-Layer Separation

Locators in src/locators/, actions in src/pages/, setup in src/fixtures/, narrative in tests/. Cross-layer leakage blocks PR merge.

accessibility_new

Semantic Locators First

getByRole, getByLabel, getByText — in that order. CSS and XPath are banned except for OutSystems OSUI widgets where no semantic option exists.

verified

Web-First Assertions

Always expect(locator).toBeVisible() — never locator.isVisible() followed by an expect(bool). Web-first assertions auto-retry and respect timeouts.

hourglass_empty

No Arbitrary Waits

waitForTimeout() and networkidle are banned for OutSystems. Wait on observable DOM state: heading visible, overlay gone, control enabled.

label

Allure Metadata Required

Every test has suite, severity, tag, and test.step() blocks. Tests without complete metadata fail lint.

do_not_disturb_on

No Cross-Test Coupling

Every test is independent. Shared state flows through setup projects and storage-state files, never through describe.serial unless explicitly designed.

bug_report

Classify Failures Correctly

An expectation mismatch is never "fixed" by weakening the assertion. Genuine product defects are marked test.fail() and escalated — the automation must reflect reality, not the bug.

account_tree

Path Aliases Only

Imports use @pages/, @locators/, @fixtures/, @helpers/. Relative paths climbing more than one level are forbidden.
