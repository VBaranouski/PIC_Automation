Tier 1 — GitHub Copilot Enterprise Agreement

Requirements Covered by SE Contract

These requirements are fulfilled by Schneider Electric's enterprise agreement with GitHub and Microsoft. No project-level action is required beyond maintaining the corporate subscription and SSO.

Req ID Requirement Status How it is covered
CTS058_001 Data confidentiality — encrypt at rest & transit Covered GitHub encrypts all data at rest (AES-256) and in transit (TLS 1.2+). Applies to all prompts, completions, and stored code.
CTS058_003 Data access control Covered Corporate SSO + SE identity governance. Only employees with an active SE Copilot licence can invoke the model.
CTS058_004 PI training data limitation Covered GitHub Copilot Enterprise contractually excludes all customer prompts and code from model training pipelines (Microsoft Enterprise Data Protection).
CTS058_012 Model cryptographic signature Covered GitHub manages model integrity, versioning, and signing for all enterprise-tier hosted models.
CTS058_013 Model redundancy & rollback Covered GitHub maintains versioned model rollback capability; enterprise customers are not exposed to breaking model changes without notice.
CTS058_019 Rate limiting on inference calls Covered GitHub Enterprise enforces per-organisation and per-user rate limits on Copilot API calls.
CTS058_021 Meta parameters enforcement (temperature, top-k, top-p) Covered GitHub controls inference hyperparameters for the hosted enterprise model. SE org admins can request policy alignment via the GitHub support channel.
CTS058_027 Restrict outbound connections Covered Copilot Enterprise model runs on GitHub-controlled infrastructure with no untrusted third-party outbound connections from the inference layer.
CTS058_028 Model deployment privileges Covered Model deployment is controlled exclusively by GitHub. SE users have no ability to modify the hosted model — eliminating this attack surface entirely.
CTS058_029 Model signature verification before execution Covered Handled by GitHub's deployment pipeline before any model is made available to enterprise customers.
CTS058_030 Model access control — authenticated users only Covered SSO-gated; only authorised SE identities can connect to the Copilot Enterprise endpoint.
CTS058_033 Data & model interaction logging Covered GitHub Enterprise audit log records all Copilot usage events. SE security team can query usage by user, repo, and timestamp.
CTS058_034 Log retention policy Covered SE's GitHub Enterprise Agreement defines log retention periods aligned with SE's data governance policy.
CTS058_035 Detection and alerting Covered GitHub security monitoring and anomaly detection runs at the platform level across all enterprise organisations.

Key guarantee: GitHub Copilot Enterprise is architecturally isolated from the public/consumer Copilot tiers. No automation prompt, PICASso business context, internal test data, or generated script is ever shared with the public model or used for training — this is a contractual, not just a policy, commitment.

Tier 2 — Project-Level Controls

Requirements Met at Repository Level

These requirements are addressed through deliberate design decisions in the PICASso automation repository — the .gitignore, .copilotignore, agent instruction files, skill architecture, and the system prompt in copilot-instructions.md. All 3 previously identified gaps have been resolved as of April 2026.

Req ID Requirement Status Implementation
CTS058_002 Data segregation (user input / system prompt / RAG data) Implemented Instruction files (.github/instructions/), skill files (.agents/skills/), and prompt files (.github/prompts/) are maintained as separate, typed artefacts. Test data never flows into instruction context.
CTS058_005 Development data sanitisation — no sensitive data fed to AI Implemented .gitignore excludes .env, config/users/, .auth/, .auth-state.json, .doc-state.json, and all state files. These are never indexed by Copilot. .copilotignore adds a second exclusion layer for test artefacts and scenario input files.
CTS058_014 AI input format control Implemented Agent instructions define strict file-type and path constraints. The knowledge-routing.instructions.md limits which files the agent reads and how (line-range surgical reads only).
CTS058_015 GenAI input/output filtering Implemented copilot-instructions.md system prompt enforces content policy alignment. Harmful, hateful, or lewd requests are rejected at the system-prompt level ("Sorry, I can't assist with that").
CTS058_016 Advanced prompt injection blocking Implemented code-review skill includes explicit prompt-injection detection instructions. The agent is instructed to alert the user and halt if tool outputs contain adversarial redirect attempts.
CTS058_022 Master prompt must not contain secrets or identifiers Implemented copilot-instructions.md and all skill/prompt files contain zero API keys, SESA IDs, database names, email addresses, or credentials. Verified by repository audit.
CTS058_023 Strong master prompt defining role, limits, capabilities Implemented copilot-instructions.md serves as the master prompt, defining the agent role (test automation engineer), scope (PICASso / Playwright), forbidden actions (no destructive ops, no security bypasses), and escalation rules.
CTS058_024 Defensive chain of thought Implemented The code-review skill implements a defensive reasoning pass on every generated change — OWASP Top 10 alignment, prompt injection scan, security bypass prohibition, and insecure pattern auto-fix.
CTS058_018 Human-in-the-loop validation Implemented The 7-step automation workflow mandates headed browser validation (Step 4) and human test run review (Step 5) before any test is marked as automated. No AI-generated code is merged without human review.
CTS058_025 Restrict data sources (Copilot indexing scope) Implemented .copilotignore added at repository root. Excludes allure-results/, allure-report/, playwright-report/, test-results/, tmp/, config/scenarios.db, docs/ai/test-cases/input/, and all auth/state files from Copilot's context window. Resolved April 2026.
CTS058_022 audit Instruction files verified free of secrets & identifiers Verified Full grep scan of .github/ and .agents/ completed. Findings: <CONFLUENCE_TOKEN> and <JIRA_TOKEN> are angle-bracket placeholder templates (not real credentials); password occurrences are locator/parameter names in code examples. Zero real secrets found. Resolved April 2026.
CTS058_021/027/036 SE IT org-level Copilot policy confirmation In Progress SE IT confirmation request submitted for: conservative temperature/top-p defaults, content policy enforcement enabled, telemetry collection disabled, org flagged as no-training, and outbound connections limited to GitHub infrastructure. Pending written confirmation from SE IT admin.
Tier 3 — Gaps & Recommended Actions

Open Items — Recommended Improvements

Three areas where the current project configuration does not fully satisfy the CTS058 intent. All are low-effort to address and do not require changes to the enterprise agreement.

ACTION 1  ·  HIGH

Add a .copilotignore file to restrict Copilot file indexing

Without a .copilotignore, GitHub Copilot indexes all non-gitignored files — including docs/ai/test-cases/ scenario inputs, allure-results/ JSON, and playwright-report/ output. These contain internal workflow identifiers and test data that should not be in Copilot's context window.

# .copilotignore allure-results/ allure-report/ playwright-report/ test-results/ tmp/ config/scenarios.db docs/ai/test-cases/input/
Addresses: CTS058_025 (Restrict data sources) · CTS058_002 (Data segregation)
ACTION 2  ·  MEDIUM

Audit all instruction & prompt files for accidental identifiers

CTS058_022 requires that the master system prompt contains no confidential data, API keys, SESA IDs, or database names. While the current files appear clean, a one-time automated scan should be run and repeated on each significant update to instruction files.

# Run from repo root — scan all agent instruction files grep -rEi \ "(sesa|@se\.com|jira\.se|confluence\.se|\ api[_-]?key|auth[_-]?token|password|secret|bearer)" \ .github/ .agents/ --include="*.md" --include="*.json"
Addresses: CTS058_022 (Master prompt content) · CTS058_005 (Data sanitisation)
ACTION 3  ·  LOW

Request SE IT to confirm Copilot org-level policy settings

Some CTS058 controls (temperature/top-p locking, content exclusion policies, outbound scope confirmation) are configurable at the GitHub organisation level by SE's IT admin. Requesting written confirmation that these are set to conservative defaults provides a formal compliance evidence trail for CTS058_021 and CTS058_027.

Specifically confirm: content policy enforcement is enabled, telemetry collection is disabled, and the org is flagged as no-training in Microsoft's system.

Addresses: CTS058_021 (Meta parameters) · CTS058_027 (Outbound connections) · CTS058_036 (Drift monitoring)

Note on N/A requirements: CTS058_007 (data poisoning detection), CTS058_009/010 (model robustness training), CTS058_011 (model sizing), CTS058_020 (computation limits), and CTS058_036 (AI drift monitoring) apply to teams building or fine-tuning AI models — not to teams consuming a hosted enterprise model. These are out of scope for the PICASso automation project and require no action.
