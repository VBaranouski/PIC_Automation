Layer 1 — AI Data Privacy

GitHub Copilot Enterprise

The AI engine powering this automation platform is GitHub Copilot Enterprise, provisioned and managed exclusively by Schneider Electric's internal IT organisation. This is not the public consumer or business tier — it is a fully isolated, enterprise-contracted deployment with privacy guarantees that extend to every prompt, file, and generated artefact.

lock

No Model Training on Company Data

Prompts, code snippets, file contents, and test outputs are never used to train or fine-tune AI models. GitHub Copilot Enterprise contractually excludes all customer data from training pipelines under Microsoft's Enterprise Data Protection terms.

block

No Data Leakage to Public Models

The enterprise deployment runs in a logically isolated environment. Schneider Electric credentials, PICASso application context, test data, and internal workflow knowledge are never exposed to shared public Copilot inference infrastructure.

corporate_fare

Schneider Electric–Managed Subscription

All licences are provisioned, audited, and controlled by Schneider Electric's IT and security teams. Access is tied to corporate SSO, with full user provisioning lifecycle (hire/transfer/offboard) managed centrally.

policy

Enterprise Content Policy Enforcement

GitHub Copilot Enterprise enforces Schneider Electric's content policies at the model level. The system prompt and agent instructions in this repository align with Microsoft content policies, preventing generation of harmful, insecure, or policy-violating code.

verified_user

Audit Trail & Compliance

All Copilot usage is auditable via GitHub's Enterprise audit log. Administrators can track which users, repositories, and workspaces triggered Copilot activity — enabling compliance reporting and anomaly detection.

hub

Isolated from Consumer Copilot

Enterprise accounts use dedicated, isolated model endpoints. The public free and Pro tiers — which have different data handling policies — are entirely separate infrastructure and are not involved in any Schneider Electric automation work.

Summary: Every AI interaction in this automation platform is covered by Schneider Electric's enterprise agreement with GitHub and Microsoft. No automation prompt, PICASso business logic, internal test data, or generated script ever leaves the enterprise boundary or contributes to public model training.

Layer 2 — Automated Security Gate

Code-Review Skill: Vulnerability Scanning

Every automation session that produces code changes runs through a dedicated security gate — the code-review skill — before any code reaches the main branch. This is not a manual checklist: it is an AI-driven, OWASP-aligned static analysis pass executed automatically as part of the standard automation workflow.

Trigger
Invoked automatically at the end of every create-auto-tests and playwright-best-practices skill session. Also triggered manually via .agents/skills/code-review/SKILL.md when reviewing any PR or diff.
OWASP Top 10
The skill checks all generated TypeScript code against the OWASP Top 10 risk categories — including injection risks, insecure direct object references, security misconfiguration, sensitive data exposure, and broken access control patterns — as they apply to test automation code.
Scope
Covers all layers: locator definitions, page object methods, fixture initialisation, test scripts, helper utilities, tracker REST endpoints, and CI pipeline scripts. No generated code is exempt from the review pass.
Enforcement
Insecure patterns are flagged and fixed immediately — not deferred or accepted with comments. The system prompt instructs the AI agent: "Any insecure code should be caught and fixed immediately." Code that passes the gate is merged; code that fails is corrected in the same session.
Prompt Injection Defence
The code-review skill includes explicit prompt-injection detection. If tool outputs or file contents contain adversarial instructions attempting to redirect the AI agent, the agent is instructed to alert the user and halt the operation rather than comply.
No Security Bypasses
The agent is prohibited from bypassing safety controls — no --no-verify flags, no discarding unfamiliar files that may be in-progress work, no weakening assertions to make tests pass. Defects are classified as defects, not worked around.

Layer 3 — Repository Security

Zero-Secret Policy on a Public Repository

The PICASso automation repository is public on GitHub. This was a deliberate architectural decision, which required that the repository be designed from day one to hold zero sensitive data — no passwords, no PATs, no environment credentials, no session tokens, no user PII of any kind.

What is Enforced by .gitignore

  • .env and all .env.* variants — environment variables and secrets never committed
  • config/users/ — all user credential files excluded from version control
  • .auth/ and .auth-state.json — persisted Playwright authentication state
  • .product-state.json, .doc-state.json — runtime state files with test data IDs
  • config/scenarios.db and config/scenarios.db-wal — SQLite database and WAL sidecar
  • playwright-report/, allure-report/, allure-results/, test-results/ — generated test artefacts
  • .playwright-mcp/ — AI agent browser session diagnostic files
  • tmp/ — local scratch files, staging scripts, diagnostic outputs
  • diagnose-*.mjs, diagnose-*.ts — temporary debug scripts
  • .DS_Store — macOS filesystem metadata

Credential Handling Architecture

  • Test user credentials are stored only in config/users/<env>.users.ts — a gitignored directory on every developer's local machine
  • .env.example is the only environment file committed — it contains placeholder values only, never real secrets
  • Playwright authentication states are generated at runtime, stored locally, and never committed
  • CI/CD pipelines inject credentials via GitHub Actions encrypted secrets — never hardcoded in workflow files
  • PATs (Personal Access Tokens) for Jira and Confluence are read from .env at runtime, never interpolated into code
  • The config/scenarios.db database contains only test metadata — no user credentials or sensitive business data
  • All Allure and Playwright reports are generated locally and served from localhost — never deployed to a public endpoint
