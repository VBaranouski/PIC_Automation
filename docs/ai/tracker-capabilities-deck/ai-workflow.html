End-to-End AI Pipeline

From Xray Case to Passing Test
in One Flow

The full lifecycle of a test scenario — from manual Xray case to AI-analyzed, automation-ready scenario, through AI-written Playwright test script, browser-validated execution, and one-click Jira defect reporting.

Phase 0 — Before Automation

Preparing Automation-Ready Scenarios

Before any test script is written, scenarios must be structured, validated, and approved. AI synthesizes knowledge from five independent sources, then a human gives the final sign-off.

1
Knowledge Source

Confluence Documentation & User Guides

AI reads feature documentation, process flows, and user guides from Confluence to understand the intended behavior, business rules, and edge cases for each feature area.

What it extracts: field validation rules, workflow state machines, role-based access constraints, UI layout expectations, and cross-feature dependencies. This forms the ground truth of what the application should do.
2
Knowledge Source

Jira User Stories & Acceptance Criteria

AI parses user stories and their acceptance criteria from Jira to capture the exact conditions that define "done" for each feature increment — including boundary conditions and negative scenarios.

What it extracts: Given/When/Then acceptance criteria, business-critical validation rules, permission boundaries, error-state expectations, and cross-story dependencies.
3
Legacy Data

Existing Test Cases (Xray / Manual)

AI analyzes the existing 4,220+ manual test cases from Xray — identifying outdated steps, vague assertions ("verify page looks correct"), and duplicates. Reusable logic is preserved; the rest is rewritten.

What it does: deduplicates overlapping cases across workflows → flags ~50% as outdated or untestable → rewrites vague steps into precise, DOM-aware assertions → carries forward valid preconditions and data requirements.
4
AI Exploration

Exploratory Testing via Playwright CLI

AI launches a real browser session through Playwright CLI and explores the live application — discovering actual page structures, element states, dynamic behaviors, and edge cases that documentation alone doesn't capture.

What it discovers: actual DOM structure and ARIA roles → OutSystems-specific widget patterns (OSUI dropdowns, partial refresh) → loading states and timing characteristics → hidden validation messages → real data values and format patterns → responsive layout breakpoints.
5
Human Gate

Human Review & Approval

A test engineer reviews the AI-prepared scenarios — verifying business logic accuracy, checking priority assignments, and confirming that the structured steps match real user workflows. Only approved scenarios enter the tracker.

Review checklist: business rule correctness ✓ → step-by-step completeness ✓ → expected results are assertion-ready ✓ → priority and feature area assignments ✓ → no duplicate coverage ✓ → approved → import to tracker via XLSX or API.
Continuous Improvement: Every step in this pipeline is built on best practices and proven techniques — and each one can be enhanced at any time with new inputs. New Confluence pages, updated acceptance criteria, fresh exploratory findings, or refined test patterns are fed back into the process, making every subsequent batch of scenarios more accurate and complete than the last.

Phase 1 — Step-by-Step

How Every Scenario Gets Automated

The 8-step process, powered by the picasso-test-automation AI skill.

1
AI + System

Scenario Ingestion & Migration

Manual test cases from Xray/Jira are exported and fed to the AI. The AI analyzes each case, identifies outdated or untestable steps (e.g. "verify the page looks correct"), and rewrites them as structured, automation-ready scenarios with precise assertions and DOM-aware steps.

What happens: AI reads the manual case → identifies outdated flow references → rewrites steps with specific locator strategies and expected values → assigns priority, workflow, and feature area → imports into tracker via XLSX or direct API. Result: scenario appears in tracker with state pending.
2
Tracker

Scenario Marked Pending

The scenario lands in the tracker with automation_state: pending. The dashboard shows it in the pending count. It's visible in workflow accordions, filterable by priority and area. Ready for the AI to pick up and script.

Dual-status model: automation_state = pending, execution_status = not-executed. The scenario has structured test steps and expected results already filled in from step 1.
3
AI Agent

AI Writes Playwright Test Script

The AI agent (using the picasso-test-automation skill) queries the tracker for pending scenarios, reads the structured steps and expected results, then generates a complete Playwright TypeScript test following the project's four-layer architecture.

Four-layer output:
1. Locator factory — semantic locators using getByRole(), getByText()
2. Page Object method — reusable actions on BasePage subclass
3. Fixture integration — scenario uses custom fixtures for auth, navigation
4. Test with Allure metadata — test.step() blocks, severity, tags, suite
4
Playwright MCP

Browser Validation via Playwright MCP

The AI opens a headed Playwright browser session and walks the full user journey defined by the test scenario. Using Playwright MCP (Model Context Protocol), the AI inspects each page element, verifies locators against the live DOM, and identifies mismatches before the test even runs.

What the AI checks: ARIA roles, accessible names, element visibility, OutSystems OSUI custom widget types (native <select> vs. OSUI dropdown), duplicate role collisions, partial-refresh content loading patterns, and conditional element states (disabled buttons, loading spinners).
5
AI Self-Correction

Fix Locators & Script Issues

When the browser validation or headed test run reveals issues — wrong locator, timing problem, OutSystems partial-refresh race condition — the AI analyzes the failure context (screenshots, error traces, DOM snapshots) and fixes the script.

Common fixes:
• Replace textContent() snapshot reads with web-first expect(locator).toContainText()
• Switch from selectOption() to click → type → pick for OSUI dropdowns
• Add exact: true or container scoping for ambiguous ARIA roles
• Add waitForOSLoad() after actions that trigger OutSystems screen reload
• This loop (steps 4–5) repeats until all locators are verified against live DOM.
6
Execution

Headless Test Run

After browser validation passes, the AI triggers a full headless Playwright run — either from the tracker UI (one-click) or via CLI. The test runs against the real QA environment with the correct Playwright project and dependencies.

Run options: per-scenario (grep), per-subsection, per-spec-file, or full suite. The tracker auto-resolves the correct Playwright project (e.g., doc-detail-offer, chromium) based on spec-to-project mapping. TypeScript compilation check (tsc --noEmit) runs alongside to catch type errors.
7
Auto-Sync

Status Update — Automated + Passed/Failed/Skipped

The tracker automatically parses the Playwright JSON report and updates the scenario: automation_state → automated, execution_status → passed. If the test fails due to a product defect, it's marked failed-defect with the failure message captured in execution notes.

Auto-sync logic: Fuzzy scenario-to-test matching links tracker scenario IDs to Playwright test titles. Timeout errors are detected and annotated. Execution notes are enriched with runtime failure messages. Spec-file mappings are updated if they changed. On-hold scenarios are automatically skipped.
8
Report + Action

Allure Report → One-Click Jira

The Allure report is auto-generated and accessible from the tracker sidebar. View pass/fail details, screenshots, execution traces. If a defect is found, submit a Jira ticket with all context pre-populated — screenshots, assertion details, environment info — in one click. No manual copy-paste.

Report contains: test name, scenario ID, step-by-step breakdown, failure assertion with expected vs. actual, screenshot at failure point, execution trace, duration, environment details, and Allure tags. All of this flows into the Jira ticket automatically.
AI Skill

The picasso-test-automation Skill

A purpose-built AI skill that encodes PICASso-specific automation best practices, OutSystems DOM patterns, and the project's four-layer architecture.

schema

OutSystems-Aware Locators

Knows that OSUI dropdowns aren't native <select>, that partial refresh replaces DOM subtrees, that CSS-generated asterisks indicate mandatory fields, and how to handle user lookup widgets. No generic tool has this knowledge.

auto_fix_high

Self-Correcting Execution

Uses Playwright MCP to open a real browser, inspect elements, and fix locators before running the test. The fix-and-retry loop continues until the script passes or a genuine product defect is identified.

layers

Four-Layer Code Generation

Every generated test follows the project architecture: locator factory → page object method → custom fixture → test with Allure metadata and test.step() blocks. Consistent, maintainable code from the start.

batch_prediction

Batch Automation

Automate 3+ scenarios at once. The skill queries the tracker for pending scenarios by feature area and priority, scripts them in sequence, validates each in-browser, and updates the tracker in bulk after all pass.

build

Failure Diagnosis

When a test fails, the AI analyzes test-results/ artifacts — error context, screenshots, traces — and categorizes the failure as a locator issue (fix it), a timing issue (add proper waits), or a product defect (mark test.fail()).

security

Role-Aware Testing

Maps scenarios to PICASso's 78 roles and 157 privileges. Generates permission-boundary tests automatically — ensuring the right users can access the right features, and unauthorized access is properly blocked.
